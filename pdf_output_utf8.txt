PhishGuard — Complete Architecture &
Implementation Plan
1. Introduction
PhishGuard is a phishing detection platform built around a browser extension that analyzes URLs,
webpages, domains, and browser activity in real time to detect phishing attacks before users submit
credentials.
The system combines:
Browser-side analysis
URL feature engineering
DNS and WHOIS enrichment
Threat intelligence
Machine learning classification
SOC-style incident workflows
Analytics and monitoring
The goal is not only to classify a URL as malicious or benign, but also to:
explain WHY the decision was made,
support incident response,
continuously improve detection quality,
and reduce false positives.
The architecture is designed to be modular so that heuristic systems can initially power detection while
machine learning models are added later .
2. Core System Objectives
The system should:
Detect phishing URLs before credential submission
Analyze webpages in real time
Detect typosquatting and domain impersonation
Perform live DNS and WHOIS enrichment
Use threat intelligence feeds
Generate explainable risk scores
Maintain SOC investigation queues
Provide analytics dashboards
Continuously retrain ML models
Minimize browser-side computational overhead

3. Complete Detection Workflow
This section explains exactly how a phishing URL moves through the system.
Step 1 — Browser Navigation Monitoring
The extension continuously monitors:
tab changes,
navigation events,
redirects,
URL changes,
form submissions.
When a user opens a webpage:
The extension captures the URL.
The URL is normalized.
Features are extracted locally.
The data is sent to the backend.
Example:
https://paypal-login-secure-update.com/verify/account
The extension should immediately separate:
protocol,
hostname,
subdomains,
TLD,
path,
query parameters.
4. URL Feature Engineering
URL features are the most important component of the initial MVP .
Feature engineering means converting URLs into measurable signals that a model can understand.

4.1 Length Features
Phishing URLs are often unusually long because attackers:
hide malicious paths,
mimic trusted brands,
add redirect parameters,
include random tokens.
Features
Feature Description
url_length Total URL length
hostname_lengthDomain length
path_length URL path length
query_length Query parameter length
Example:
https://paypal.verify.account.security-update.ru/login/verify/user/account
Long URLs are suspicious but not automatically malicious.

4.3 Typosquatting Features
Typosquatting is when attackers create domains similar to legitimate brands.
Examples:
paypa1.com
faceb00k.com
google-auth-login.net
Levenshtein Distance
Measures how many edits are required to transform one string into another .
Example:
paypal -> paypa1
Distance = 1
Small edit distance from major brands is highly suspicious.
Homoglyph Detection
Attackers replace characters with visually similar ones.
Example:
rnicrosoft.com
Looks visually similar to:
microsoft.com
Unicode normalization should be performed before analysis.

4.5 DNS Features
The backend performs live DNS lookups.
DNS Information Collected
Data Purpose
A record Maps domain to IP
MX recordMail infrastructure
NS record Nameservers
TTL DNS behavior
ASN Hosting provider reputation
Fast Flux Detection
Fast flux is when attackers rapidly rotate IP addresses.
Indicators:
rapidly changing DNS records,
multiple unrelated IPs,
abnormal TTL values.
This helps attackers evade blacklists.

18. Final Engineering Notes
Phishing detection is fundamentally a probabilistic security problem.
No single indicator proves maliciousness.
A strong system combines:
heuristics,
machine learning,
threat intelligence,
browser analysis,
authentication signals,
and human analyst feedback.
The biggest engineering challenge is balancing:
detection rate,
latency,
explainability,
and false positives.

For an MVP , prioritize:
reliable feature extraction,
fast URL analysis,
explainable scoring,
stable backend architecture.
Do NOT begin with deep learning.
Most successful phishing systems initially succeed because of:
good feature engineering,
strong heuristics,
quality enrichment,
and feedback loops.
Machine learning improves the system later , but architecture and data quality determine whether the
platform succeeds.
